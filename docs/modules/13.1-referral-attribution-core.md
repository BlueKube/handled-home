## 0) Why this module exists

Handled Home’s viral growth depends on one thing: **trustworthy attribution**.

If we can’t answer “who referred whom?” and “did they earn it?”:
- incentives get abused  
- payouts get disputed  
- ops gets dragged into manual fixes  
- growth becomes human-bottlenecked

Module 13.1 creates a deterministic, idempotent referral + reward system that:
- survives webhook duplicates/out-of-order delivery  
- links cleanly to Module 11 ledgers  
- can be reused by Founding Partners, customer referrals, provider recruiting, and market cohorts

---

## 1) North Star outcomes (Definition of Done)

1) Any signup can be attributed via link/code **once** (first-touch by default).  
2) Referral milestones are captured deterministically:
   - signup → subscribe → first completed visit → first paid cycle  
3) Reward records are created **exactly once** per milestone and program.  
4) Rewards are paid automatically via Module 11:
   - customer credits → invoice line items  
   - provider bonuses → payouts  
5) Fraud controls exist at the data layer:
   - holds, caps, risk flags, dedupe  
6) Admin can audit any reward end-to-end in < 60 seconds:
   - referral → milestones → reward → ledger event → invoice/payout

## 3) Core concepts

### 3.1 Program
A set of rules that define:
- who can refer  
- who can be referred  
- milestones that trigger rewards  
- reward types and amounts  
- holds/caps/fraud gates  
- messaging templates (share copy)

### 3.2 Referral relationship
A unique linkage between:
- a referrer (customer user or provider org)  
- a referred user (after signup)  
- a program + cohort

### 3.3 Milestone
A verifiable system event. Examples:
- `subscribed` (subscription active + first payment successful)  
- `first_visit` (first completed job)  
- `paid_cycle` (first subscription invoice paid)

### 3.4 Reward
A payable unit tied to a milestone:
- customer credit (applies to next invoice)  
- provider bonus (paid via payout run)  
Rewards are **not** paid on vanity metrics like “invite sent.”

---

## 4) Attribution rules (deterministic + simple)

### 4.1 Attribution sources
- referral link (customer or provider)  
- referral code entry (manual)  
- admin import (sales-assisted)  

### 4.2 Default attribution policy
- **First-touch wins** (simplest, least disputable).  
- If a user arrives later via another link, ignore it.

### 4.3 Attribution time window (recommended)
- Referral cookie/session window: configurable (e.g., 30 days)  
- If user signs up after window:
  - still allow code entry to attribute if provided

### 4.4 Conflict handling
If user already attributed:
- do not overwrite  
- allow **admin override** only:
  - requires reason  
  - audit logged  
  - shows before/after

---

## 5) Milestone capture (system-driven truth)

### 5.1 Milestones (MVP set)
Customer milestones:
- `installed` (signup complete; optional, low value)  
- `subscribed` (subscription becomes active)  
- `first_visit` (first completed job)  
- `paid_cycle` (first invoice paid)  

Provider milestones (used by 13.2 but captured here):
- `provider_ready` (payout account READY)  
- `provider_first_job` (first job completed)  

### 5.2 Canonical sources (avoid ambiguity)
- `subscribed`: Module 05 + Module 11 payment success  
- `first_visit`: Module 09 job completion (server-validated)  
- `paid_cycle`: Module 11 invoice paid webhook  
- `provider_ready`: Module 11 payout onboarding status  

### 5.3 Idempotent milestone creation
Rule:
- `(referral_id, milestone)` is unique  
- duplicates (webhook retries) are ignored safely

---

## 6) Reward engine (ledger-first)

### 6.1 Reward types (MVP)
- `customer_credit` → creates Module 11 customer ledger event and invoice line item  
- `provider_bonus` → creates Module 11 provider ledger event, included in payout runs  
- `revshare` (optional, admin-only, time-boxed; can be deferred)

### 6.2 Reward creation rules
- Created only when milestone occurs  
- Created only if program rules allow for that role/zone/category  
- Created idempotently:
  - unique `(program_id, referred_user_id, milestone, reward_type)`

### 6.3 Reward lifecycle
- `pending` → `on_hold` → `earned` → `applied/paid` → `closed`  
Customer credit:
- `applied` when attached to next open invoice  
Provider bonus:
- `paid` when payout webhook confirms

### 6.4 Holds + fraud gates (default posture)
- Most rewards should be held until strong milestones:
  - customer credits: hold until first successful payment (or first visit)  
  - provider bonuses: hold 7–14 days or until first visit  
- High-risk patterns trigger `on_hold` + admin exception

---

## 7) Fraud & abuse controls (data-layer)

### 7.1 Dedupe rules (must-have)
- one referred user → one referrer (per program)  
- one milestone → one reward  
- prevent “self-referral” where possible:
  - shared phone/email/device heuristics (soft flag, not hard block)

### 7.2 Caps (policy-driven)
- max rewards per referrer per week  
- max total reward dollars per referrer per 4 weeks  
- max auto credits per customer per 4 weeks (align with Module 11/12)

### 7.3 Risk flags + exception routing
If suspicious:
- mark reward `on_hold`  
- create referral risk flag  
- push to admin queue (integrate with `/admin/exceptions` from Module 11)

## 9) Admin UX (critical)

### 9.1 Programs
- create/edit referral programs  
- set milestone triggers and reward amounts  
- set holds and caps  
- set templates

### 9.2 Audit view
Admin can click any reward and see:
- program + cohort  
- referral relationship  
- milestones achieved  
- reward status + hold reason  
- ledger reference (invoice/payout id)  

### 9.3 Overrides (rare)
- override attribution

## 11) RLS & security

Customers:
- read only their own referral codes/referrals/rewards  
- cannot modify milestones/rewards

Providers:
- read only rewards tied to their provider_org  
- cannot modify reward state

Admins:
- full access; overrides logged

All “apply reward” actions must be server-side RPCs.

---

## 12) Acceptance tests (explicit)

1) Signup via referral link attributes user once (first-touch).  
2) Subscription becomes active → milestone created once.  
3) First job completed → milestone created once.  
4) Reward created once and placed on hold per policy.  
5) Reward released → customer credit appears on next invoice.  
6) Provider bonus flows into payout run and is paid once on webhook.  
7) Duplicate webhooks do not duplicate milestones/rewards.  
8) Admin can audit reward end-to-end and override with reason.

Module 13.1 is done when attribution, milestones, rewards, holds, and ledger-linking are **correct, idempotent, and auditable**—so Modules 13.2 and 13.3 can scale virally without humans becoming the growth bottleneck.
