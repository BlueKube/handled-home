## 0) North Star outcomes (Definition of Done)

1) Shares happen **without begging**: the product generates naturally shareable artifacts after real value moments.  
2) Growth is measurable: every viral action is captured via an append-only event bus with idempotency.  
3) Autopilot can steer growth by surface: it can boost/quiet prompts per zone/category state.  
4) Fraud resistance: viral loops can’t be used to farm credits or bonuses.  
5) Provider promise control: share scripts never imply custom scheduling or off-policy outcomes.  
6) Organic-first: early virality is **proof-led**, not incentive-led.  
7) Privacy-first: share links are safe by default (anonymous, limited location, expiring).  
8) Premium brand: sharing is **high-quality**, not spammy.

## 1) Product strategy: “3 viral surfaces, one engine”

Handled Home should not rely on a single loop. It should run three, in this order of launch impact:

### Surface A — Customer “Handled Receipt Share Card” (primary)
**Why:** customers share proof + transformation more readily than they share “download this app.”  
**Trigger:** after a completed job receipt is generated.  
**Default format (organic-first):** **single ‘after’ hero shot** (privacy-safe) + subtle stamp.  
**Optional format:** before/after toggle (user-initiated) with preview.  
**Content:** best after photo + “Handled” stamp + date + category + minimal brand mark.

### Surface B — Provider “Invite from Payout / Milestone” (secondary)
**Why:** providers share when dopamine + money is present, and the action is one tap.  
**Trigger:** payout posted, bonus earned, milestone progress (“2 customers away from Founding Partner tier”).  
**Content:** magic link + category/state-safe scripts.

### Surface C — Customer “Invite your other pro” (cross-pollination)
**Why:** each household uses multiple recurring pros; this loop can accelerate category expansion.  
**Trigger:** after subscription activation OR after first successful visit.  
**Content:** pick one other category → prewritten SMS to invite their pro.

Autopilot (13.3) should be able to weight these surfaces by market health.

**Default launch posture:** Surface A strong; Surface B moderate; Surface C light.

## 2) Hosting strategy for share landing pages (critical)

### 2.1 Recommended: separate lightweight public site
Host share landing pages on a **separate lightweight public site** (e.g., `share.handledhome.com`) rather than inside the main app bundle.

**Why:** faster loads, reliable social previews/OG tags, no auth/app boot overhead, easier A/B testing and iteration.

### 2.2 Implementation approach
Share link uses a `share_id`:
- Landing page loads instantly
- Fetches a **public, privacy-safe payload** via a read-only endpoint:
  - after image URL (optimized/resized; CDN-friendly)
  - category + date
  - 1–3 bounded checklist bullets
  - market availability state (open vs waitlist)
- CTA deep links:
  - if installed: open app
  - else: store install

## 3) Share link lifetime + revocation policy (privacy-first)

### 3.1 Defaults
- **Default expiry:** 30 days
- **Optional user choice:** “Make this link permanent” (explicit consent)
- **Hard cap:** even “permanent” links auto-expire at **12 months** unless renewed
- **Revocation:** customer can disable any share link from the receipt (“Disable shared link”)

### 3.2 Asset access must match link access
Expiring the page is not enough—also expire/revoke image access:
- store share assets in a dedicated bucket/folder
- use token-gated access or signed URLs with rotation
- ensure CDN caching respects revocations (cache-bust on revoke)

## 4) The Growth Event Bus (required for true Autopilot)

### 4.1 Why we need it
13.1 has deterministic attribution; 13.3 has state decisions; but “automatic virality” needs:
- consistent instrumentation across modules
- surface-level conversion tracking (prompt → share → click → install → subscribe → paid cycle)
- reliable dedupe for retries and offline events

### 4.2 Data model (Supabase)

#### `growth_events` (append-only, canonical)
- `id` uuid pk  
- `created_at` timestamptz  
- `event_type` text (enum-like)  
- `actor_role` text (`customer|provider|system|admin`)  
- `actor_id` uuid  
- `zone_id` uuid null  
- `category_id` uuid null  
- `sku_id` uuid null  
- `cohort_id` uuid null (13.3)  
- `source_surface` text  
  (`receipt_share_card|provider_milestone_share|cross_pollination_invite|provider_invite|landing_page|other`)  
- `context` jsonb  
- `idempotency_key` text unique  

`context` examples:
- `{ job_id, receipt_id, before_after_asset_ids, invoice_id }`
- `{ share_id, share_expiry_mode: default_30d|user_permanent|hard_cap_12m }`
- `{ market_state, prompt_variant, copy_variant }`
- `{ share_channel: sms|whatsapp|copy_link|qr|native_share }`
- `{ share_brand_mode: minimal|full }`
- `{ share_asset_mode: after_only|before_after }`
- `{ landing_variant }`

#### `growth_surface_stats_daily` (materialized or scheduled)
Aggregates per (zone, category, surface):
- prompt_shown
- share_initiated
- share_completed
- landing_viewed
- store_click
- signup
- subscribed
- first_visit
- paid_cycle

### 4.3 Event types (MVP)
- `prompt_shown`
- `share_initiated`
- `share_completed`
- `landing_viewed`
- `store_clicked`
- `invite_sent`
- `invite_opened`
- `signup_completed`
- `subscription_started`
- `first_visit_completed`
- `paid_cycle_completed`

Server-side event creation preferred; client events must be idempotent.

## 5) Surface A: Receipt Share Cards (Customer) — Organic-first

### 5.1 UX entry points
- On receipt (Module 10): CTA **“Share the after photo”**
- On dashboard: “Your latest result” card with share CTA
- Optional: “Add before/after” toggle (secondary action)
- Never prompt if a dispute exists for that job.

### 5.2 Default share asset (privacy-safe)
Default is **AFTER-ONLY**:
- one best after photo (AI-selected)
- subtle stamp: **“Handled”** + date + category
- minimal brand mark (small logo mark only; no CTA)
- no address; no exact neighborhood by default

Optional controls:
- toggle: “Add before/after” (preview first)
- toggle: “Show my first name” (off by default)
- toggle: “Show neighborhood” (off by default; coarse only)

**Quality gate:** if photo quality is weak, do not prompt customer to share.

### 5.3 Branding modes
- Default: **minimal branding** (“Handled” stamp + tiny mark)
- Optional: “Show Handled Home” (adds full brand name; still no incentive)

### 5.4 AI add-on: best photo selection (allowed)
AI may:
- select best after (and best before if toggled)
- detect low-quality / too-similar photos
- create a provider coaching hint for future jobs

AI must not:
- claim outcomes not evidenced
- add deceptive overlays

### 5.5 Prompt frequency caps (anti-annoyance)
- max 1 share prompt per job completion
- max 1 reminder per week
- disable prompts for flagged/high-risk accounts or after negative feedback

## 6) Landing page experience (“Wow” page)

### 6.1 Default: lightweight wow page
Single-screen page:
- full-width after photo
- “Handled.” + category + date
- one-line promise: “Recurring home services, automatically handled.”
- primary CTA: **Get Handled Home**
- secondary CTA: **I’m a provider**
- availability state-aware (open vs waitlist)

### 6.2 Anonymity default
Landing is **anonymous by default**:
- headline: “A Handled result.”
Optional opt-in: “Shared by {first_name}” (toggle off by default)

### 6.3 Micro-checklist (trust builder)
Include 1–3 bullets from SKU/checklist:
Example: “Exterior windows” • “Sills wiped” • “Screens rinsed”  
No custom text. No address exposure.

---

## 7) Provider sharing policy (premium + consent-safe)

### 7.1 Default: provider cannot share customer cards
To prevent spam and consent issues, providers do **not** get share access by default.

### 7.2 Controlled, earned provider sharing (optional v2)
If enabled later, require ALL:
1) Customer opt-in per job: “Allow provider to share this result” (off by default)  
2) Sanitized asset (no name, no location)  
3) Provider in good standing (low dispute rate; no recent safety/behavior flags)  
4) Hard rate limit (e.g., 3 shares/week)  
5) Providers share a **provider portfolio card** (generic), not the customer’s share link.

### 7.3 Provider portfolio card (safer)
A generic “Handled result” card for provider marketing:
- shows after photo only
- no customer identity
- links to generic category landing (recommended early) or provider profile (later)

## 8) Surface B: Provider Milestone / Payout Share (Provider)

### 8.1 Triggers
- payout posted (Module 11)
- bonus earned (13.1 rewards)
- progress thresholds (13.2): “2 customers away…”

### 8.2 One-tap invite
Provider sees:
- CTA: **Invite customers**
- Copy link + scripts
- optional QR

### 8.3 State-aware scripts (must-have)
Dynamic by market state:
- OPEN: “We’re live now…”
- SOFT_LAUNCH: “Limited early spots…”
- WAITLIST/CLOSED: “Launching soon…”

No promises of specific times, custom scheduling, refunds.

### 8.4 Provider spam prevention
Rate limiting + consent confirmation + autopilot throttles if abuse detected.

### 9.1 Trigger moments
- after subscription activation
- after first completed visit (high trust moment)

### 9.2 UX flow
Pick another category → “Text my pro” / “Send to neighbor” → 2 taps.

### 9.3 Guardrails
State-aware availability; high-risk accounts get fewer prompts; incentives off by default.

## 10) Autopilot steering (13.3 integration)

Add dials:
- `surface_weights`: {receipt_share, provider_share, cross_pollination}
- `prompt_frequency_caps`
- `incentive_visibility`: default OFF
- `share_brand_default`: minimal|full (default minimal)
- `share_link_expiry_default_days`: default 30
- `share_link_hard_cap_days`: default 365

All changes logged.

---

## 11) Fraud & abuse controls (13.1 + Module 12 synergy)
Detect:
- many invites without paid cycles
- repeated disputes from same address/device cohort
- abnormal dispute/credit rates in referred cohorts

Controls:
- delayed payouts/holds
- probation policies (tighten instant credits)
- throttle prompts and disable share tools for flagged actors
- tighten Module 12 policies for high-risk cohorts

---

## 12) Acceptance tests
1) Receipt share default is after-only; before/after optional; events emitted with idempotency.  
2) Landing page is wow-page, anonymous by default, with optional first-name opt-in.  
3) Links expire in 30 days by default; revoke works; assets become inaccessible.  
4) Provider sharing disabled by default; if enabled, requires customer opt-in and rate limit.  
5) Autopilot adjusts surface weights and caps; logs actions.  
6) High-risk cohort triggers prompt throttles + tighter dispute policies.

---

## 13) Definition of done
- Share landing pages are fast and reliable (separate site)
- Organic share is default; incentives off
- Expiration/revocation works end-to-end
- Event bus drives measurable funnels
- Autopilot can steer surfaces safely
- Provider sharing is consent-safe and off by default
