## 0) Why 06.1 exists

Module 06 is the first time Handled Home makes a **real operational promise**.  
To keep that promise premium, the system must be:
- resilient under concurrency (multiple signups at once)  
- predictable under edge cases (expired offers, abandoned rejects)  
- audit-safe (overrides and forced capacity)  
- calm for customers (no confusing “something changed” moments)  

---

## 1) Hardening goals (Definition of Done)

1) Capacity reservations are **race-safe** using Postgres row locks.  
2) Offers expire and automatically release capacity via a scheduled cleanup job.  
3) Alternative ordering is predictable and configurable (window-first vs day-first).  
4) Admin can see **why** zones are tight (override rate, rejection rate, offer expiry).  
5) Customer messaging never feels like a bug:
   - “Your offer expired, we re-matched you” (calm, helpful)

## 2) Concurrency & atomicity (Postgres-first)

### 2.1 Locking strategy (recommended)
When reserving capacity for `(zone_id, day_of_week, service_window)`:

- Use a single row in `zone_service_day_capacity` as the lock point.  
- Reservation transaction:
  1) `SELECT ... FOR UPDATE` the capacity row  
  2) compute `effective_limit = max_homes * (1 + buffer_percent/100)`  
  3) if `assigned_count < effective_limit`: increment + commit  
  4) else: fail  

This prevents two signups from incrementing into the same last slot.

### 2.2 Idempotency keys (avoid double-offers)
For `create_or_refresh_service_day_offer(property_id)`:
- If an `offered` assignment already exists and `reserved_until > now()`, return it.  
- If `confirmed` exists, return it (do not create new).  
- If expired, supersede old offer and create a new one.

### 2.3 “Move reservation” safety (alternative selection)
When selecting an alternative:
- Do it in a single transaction:
  1) lock old capacity row, decrement  
  2) lock new capacity row, increment (only if capacity allows)  
  3) update assignment day/window  
  4) mark accepted offer  
  5) commit  
- If new capacity cannot be reserved:
  - rollback entirely
  - return error: “That option just filled—pick another.”

## 3) Offer expiry & cleanup automation

### 3.1 Offer TTL (explicit rule)
- `reserved_until = created_at + 48 hours` for `status='offered'` assignments.

### 3.2 Cleanup job (required)
Run every hour (Supabase scheduled function or cron):
- Find expired offers:
  - `status='offered' AND reserved_until < now()`
- For each:
  1) release capacity for the reserved day/window  
  2) mark assignment `superseded`  
  3) (optional) write event `service_day_offer_expired`  
- Do **not** delete rows (keep history).

### 3.3 Customer experience when expired
If customer opens `/customer/service-day` after expiry:
- show:
  - “Your previous offer expired, so we refreshed your Service Day options.”
- then immediately show new offer (generated on load).

Guardrail:
- Never blame the customer.
- Always use calm language.

## 4) Alternative ordering: “window-first” vs “day-first”

Zones may have window configurations (`am`, `pm`, `any`).  
Admins should be able to set a zone preference:

- `alternative_strategy = window_first | day_first` (default `window_first`)

### 4.1 Window-first strategy (default)
When default day is full:
1) try same day other window(s)  
2) then adjacent days, same window  
3) then adjacent days, other windows  

Why:
- keeps routes densest; smallest change for customer.

### 4.2 Day-first strategy
When default day is full:
1) try adjacent day(s), same window  
2) then adjacent day(s), other windows  
3) then same day other windows (last)

Why:
- some markets have “day matters more than time window” preferences.

Implementation:
- store on `zones` (or `zone_service_day_capacity`) as `text` enum.

## 5) Admin ops upgrades (make it feel like a control panel)

### 5.1 Zone health panel (expanded)
On `/admin/service-days` zone detail show:
- Utilization % per day/window  
- Offer expiry count (last 7 days)  
- Reject rate (last 30 days)  
- Override count (last 30 days)  
- “Tightness reason” string:
  - “High utilization”  
  - “High overrides”  
  - “High rejection”  

### 5.2 Bulk actions (safe)
Admin can:
- “Extend offer TTL to 72h for this zone” (temporary)  
- “Pause new offers for this zone” (does not break confirmed customers)  

### 5.3 Override audit UX
Override modal must include:
- reason dropdown (required)  
- freeform notes (optional)  
- capacity warning banner if exceeding  
- “Force anyway” typed confirm  

All of this writes:
- `service_day_override_log`
- event in `subscription_events` or a dedicated `ops_events` table (optional)

## 6) AI / automation enhancements (safe + practical)

These features are *not* “chatbot AI”. They are guardrailed automation.

### 6.1 Explainable assignment templates (upgrade)
Add a reason code + template mapping:

Reason codes:
- `default_available`
- `default_full_used_nearby`
- `window_swap`
- `capacity_tight_balancing`

Templates:
- `default_available`:  
  - “We chose **{day}** because it’s the standard route day for your neighborhood.”
- `default_full_used_nearby`:  
  - “We chose **{day}** to keep your service consistent while your zone’s main day is full.”
- `window_swap`:  
  - “We kept **{day}** and adjusted the time window to keep routes tight and reliable.”

Admin control:
- toggle templates on/off per zone.

### 6.2 “Offer confidence” badge (customer-facing, optional)
Show a small badge:
- “Stable day” if utilization < 85%  
- “Popular day” if 85–95%  
- Hide badge if >95% (avoid anxiety)

Guardrail:
- never show “at risk” messaging to customers.

### 6.3 Drift alerts (admin) with thresholds
Trigger weekly:
- if overrides > X OR rejects > Y OR expiries spike  
Show a single banner with recommended actions:
- “Consider increasing max homes on Tue”
- “Consider adding AM window capacity”

Guardrail:
- advisory only; no auto-changes.

### 6.4 “Smart alternative mix”
When generating 3 options:
- include at least:
  - 1 near-day option
  - 1 different-window option (if windows exist)
  - 1 “best availability” option
This increases acceptance without feeling like a calendar.

### 6.5 Preference capture (customer, non-binding)
If customer rejects and can’t find an option:
- allow selecting preferred days (checkbox list Mon–Fri)
- store in:
  - `service_day_preferences` table OR jsonb on assignment
Use later for support and expansion, not promises.

## 7) Data model changes (minimal additions)

### 7.1 Additions to `zones` (or `zone_service_day_capacity`)
- `alternative_strategy` text not null default 'window_first'
- `offer_ttl_hours` int not null default 48

### 7.2 Optional new table: `service_day_preferences`
- `id` uuid pk
- `customer_id` uuid not null
- `property_id` uuid not null
- `preferred_days` text[] not null default '{}'
- `created_at` timestamptz default now()

RLS:
- customer can read/write own
- admin can read all

## 8) Additional acceptance tests (06.1)

1) Concurrent signups do not exceed capacity (verified with row locks).  
2) Expired offers release capacity automatically via scheduled job.  
3) Customer opening an expired offer sees refreshed offer with calm messaging.  
4) Alternative ordering follows zone `alternative_strategy`.  
5) Selecting an alternative is transaction-safe (no partial decrement/increment).  
6) Admin zone detail shows reject/override/expiry counts without error.  
7) Preference capture stores days and is visible to admin (optional).
